## 작업 내용
- 회원가입/계좌연결 시 클라이언트에 하드코딩된 인증코드(`1234`)로 통과시키던 가짜 KYC를 제거하고, 서버가 무작위 4자리 코드를 발급·검증하는 실제 1원 인증 플로우로 교체
- `KycVerification` 엔티티/리포지토리 추가 — 계좌번호·은행명·예금주명·코드·만료시각·시도횟수를 저장
- `POST /api/v1/mockbank/kyc/start`, `/kyc/inquiry`, `/kyc/verify` 3개 엔드포인트 추가
  - `start`: 코드 발급, 입금자명을 `fxflow{code}`로 가정
  - `inquiry`: 은행명/계좌번호/예금주명으로 입금자명 조회 (로그인 없이도 호출 가능하도록 permitAll 처리)
  - `verify`: 코드 검증 성공 시 기존 `linkAccount()`까지 한 트랜잭션으로 완료 (모의계좌+지갑 생성, KYC 완료 처리)
- 인증코드 5분 TTL, "1원 다시 요청" 시 이전 PENDING 건 자동 만료 처리
- 1원 인증 요청(최초+재요청 합산) 하루 5회 제한 — 스키마 변경 없이 `created_at` 기준 카운트 쿼리로 구현
- 프론트엔드: 회원가입(`signup`)·계좌연결(`onboarding/link-account`) 페이지에 카운트다운 타이머, "1원 다시 요청" 버튼, 오늘 남은 요청 횟수 표시 추가. 계좌번호 조회 전용 페이지(`mockbank/kyc-inquiry`) 신규 작성

## 작업 목적
- 기존 KYC는 프론트에서 `code === "1234"`만 비교하는 데모용 로직이라 백엔드가 실제로 본인 확인을 검증하지 않는 상태였음. 실서비스 흐름에 맞춰 서버가 코드를 발급/보관/검증하도록 책임을 옮겨 실질적인 계좌 소유 확인이 되도록 함
- 인증코드를 무제한 재요청할 수 있으면 코드 탐색(brute-force)에 취약하므로 하루 요청 횟수 제한 추가
- 검증 실패 시 시도 횟수를 추적해 사용자가 남은 기회를 인지하게 함으로써 무차별 입력 시도를 방지

## 리뷰 포인트
- **트랜잭션 롤백 이슈**: `verifyKyc()`는 `@Transactional`인데 코드 불일치 시 예외를 던지면 트랜잭션이 롤백되어 `attemptCount` 증가분도 함께 사라지는 버그가 있었음. 실패 횟수만 `KycAttemptService`에서 `Propagation.REQUIRES_NEW`로 별도 커밋하도록 분리해서 해결 — 이 부분 설계가 적절한지 리뷰 받고 싶음
- `BusinessException`에 동적 메시지를 받는 생성자를 추가하고 `GlobalExceptionHandler`가 `code.getMessage()` 대신 `e.getMessage()`를 쓰도록 변경함. 기존 호출부는 동일한 메시지를 반환하므로 영향 없다고 판단했으나, 다른 도메인에서도 이 패턴을 쓰는 곳이 있는지 확인 필요
- 1원 인증은 실제 자금 이동(LedgerEntry)이 전혀 없는 순수 연출(narrative)로 구현함 — 회사 풀(KRW/USD 리밸런싱)과는 무관하게 설계. 추후 감사 추적이 필요해지면 보강 필요
- `kyc/inquiry`는 비로그인 상태에서도 호출 가능하게 `SecurityConfig`에 permitAll 추가함 — 계좌번호+예금주명만으로 조회되는 구조라 USD 수취조회(`inquiry/usd`)와 동일한 보안 수준으로 판단
